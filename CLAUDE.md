# CLAUDE.md - 练了吗项目 AI 编程规范

> 源自 Andrej Karpathy 的 AI 编程最佳实践 + 第一性原理优化，针对练了吗 Flutter 项目定制

---

## ⚡ 第一性原理框架（首要原则）

**「以第一性原理！从原始需求和问题本质出发，不从惯例或模板出发。」**

| 序号 | 核心原则 | 具体操作指引 |
| :--- | :--- | :--- |
| **1** | **需求澄清机制** | 不假设用户目标清晰，动机模糊时主动讨论 |
| **2** | **路径优化建议** | 目标明确但路径非最优时，直接提供更优方案 |
| **3** | **根因分析要求** | 遇问题需追溯根本原因，避免"打补丁"式解决方案，每个决策需回答"为什么" |
| **4** | **信息精简输出** | 聚焦核心重点，剔除不影响决策的冗余信息 |

---

## 原则一：编码前先思考

**问题**：AI 容易擅自猜测需求、默认选择复杂方案

**具体规范**：

1. **不确定时必须询问，禁止猜测**
   - 如果需求有歧义，先澄清再动手
   - 例如：用户说"加个验证功能"，必须问清楚验证什么、验证规则、严格程度

2. **存在多种理解时列出选项供用户选择**
   - 不要替用户做技术决策
   - 例如：状态管理方案有多个选项时，列出优缺点让用户选

3. **发现有更简单的方案时主动提出**
   - 如果能想到更简单的实现方式，要说出来
   - 该推回来就推回来

## 原则二：简约至上

**问题**：AI 容易过度设计、添加未要求功能

**具体规范**：

1. **不编写未被要求的功能**
   - 用户要什么就做什么，不要"画蛇添足"
   - 例如：只需要一个按钮，就不要顺便加登录认证

2. **只用一次的代码不建抽象层**
   - 避免过早抽象
   - YAGNI（You Aren't Gonna Need It）原则

3. **不添加无人要求的"灵活性"和"可配置"**
   - 不需要为 hypothetical 的未来需求预留扩展点

4. **不对不可能发生的异常场景做错误处理**
   - 检验标准：一个资深工程师看了会不会说"太复杂了"。会的话，直接砍。

## 原则三：精确编辑

**问题**：AI 容易擅自重构无关代码、修改风格

**具体规范**：

1. **只动被要求动的部分**
   - 改 bug 就只改 bug，不要顺手重构旁边的代码
   - 不要修改与需求无关的变量名、注释、代码风格

2. **严格匹配项目已有的代码风格**
   - 哪怕你觉得自己写得更好，也要保持一致
   - 参考项目现有代码的 import 顺序、命名规范、格式化方式

3. **看到不相关的问题，提一嘴就行，别动手**
   - 如果发现其他 bug 或问题，可以提出来，但不要主动修改

4. **只清理自身改动导致的无用代码**
   - 如果你的改动导致某些代码不再被使用，清理掉是你的责任
   - 但之前就存在的问题，没人让你改就不要碰

## 原则四：目标驱动

**问题**：AI 依赖具体步骤指令、缺乏验证

**具体规范**：

1. **提供验收标准而非实现步骤**
   - 别告诉 AI "写一个函数实现 X"，改成"先写测试用例，然后让所有测试通过"
   - 要修 bug，改成"先写一个能复现这个 bug 的测试，然后让它通过"

2. **优先编写测试用例再实现功能**
   - TDD 原则：测试先行

3. **复杂任务需先列出带验证方式的计划**
   - 验收标准定得越清晰，AI 能独立循环执行的时间就越长，你需要介入的频率就越低

## 练了吗项目特定规范

### 技术栈
- **框架**：Flutter
- **状态管理**：GetX (路由) + 原生 setState
- **存储**：SharedPreferences
- **通知**：flutter_local_notifications

### 项目结构约定
```
lib/
├── main.dart                 # App入口
├── models/                   # 数据模型
├── screens/                  # 页面
├── services/                 # 服务层
├── theme/                    # 主题
├── utils/                    # 工具类
├── controllers/              # 控制器
└── widgets/                  # 可复用组件
```

### 代码风格
- 使用 `const` 构造函数（尽可能）
- 中文注释
- 遵循项目现有 import 规范

### Git 工作流
- 创建独立分支进行开发
- 定期 pull 保持最新
- 重要文件修改前先通知
- 合并前确保 flutter analyze 通过
